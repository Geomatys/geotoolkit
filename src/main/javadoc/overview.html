Overview of Geotoolkit.

Requirements

Geotoolkit requires Java 6 standard edition. There is additional requirement for metadata and referencing modules. But the coverage module requires the Java Advanced Imaging extension.

A port to Java 5 is planed. This port will have all JAXB annotations removed and will loose the benefit of some Java 6 methods that may help performance ({@link java.util.Arrays#copyOf(double[],int) Arrays.copyOf(...)}, {@link java.lang.Math#copySign(double,double) Math.copySign(...)}, etc.) or precision ({@link java.awt.geom.Path2D}, {@link java.awt.geom.AffineTransform#quadrantRotate(int)}, etc.).

Javadoc annotations

  • Versions

    The {@code @since} javadoc tag gives the Geotoolkit version when the annotated element (class, method, etc.) was first introduced. If the declared version is the one under development (for example if an element is annotated with {@code @since 3.0} and the current library version is Geotoolkit 3.00-SNAPSHOT), then that element is uncommited and may change in any future version - it will become stable only when the {@code -SNAPSHOT} keyword is removed from the library version number, or when the version increase.

    Stable API (for example those annotated as {@code @since 2.4}) will not change lightly. If a stable API need to be removed, it is first annotated as {@code @deprecated} and will be removed only in a future release.

    The {@code @version} javadoc tag gives the last Geotoolkit version number when the annotated element got significant changes.

  • Authors

    The code were contributed by authors from different organisations. The organisation name appears between parenthesis after the author name, sometime in abbreviated form. More than one organisation may be listed for the same author if he continued to work on the code while moving to an other organisation. The organisation names used in this documentation are:

    Geomatys  Geomatys.
    Geosolutions  Geosolutions.
    HSRS  Help Service Remote Sensing.
    IRD  Institut de Recherche pour le Développement.
    MPO  Ministère des Pêches et Océans Canada.
    OpenGeo  OpenGeo.
    Refractions  Refractions Research.
    TOPP  The Open Planning Project.
    UBC  University of British Columbia.
    UWB  University of West Bohemia.
    USGS  U.S. Geological Survey. Actually not a direct contributor to Geotoolkit, but their C/C++ code in the Proj4 project were used as a basis for the map projections.

Policies

  • Classes naming

    Implementations of GeoAPI interfaces usually (but not always) begin with {@code Default} or {@code Abstract} prefix. The later is used when a class is abstract according ISO specification; the class is not necessarly abstract in the Java sense. For example {@link org.geotoolkit.referencing.cs.AbstractCS} is abstract according ISO (users should always specify a more accurate subclasses), but can still be instantiated in Java. Actually it is instantiated in some exceptional cases (especially during WKT parsing) when the actual coordinate system type is really unknown.

    The {@code Default} or {@code Abstract} prefix cover the majority of cases for GeoAPI implementations, but there is a few exceptions. Implementations working only in space having a fixed number of dimensions are suffixed with {@code 1D}, {@code 2D}, {@code 3D} or {@code 4D}. The general implementation - if any - is prefixed with {@code General}. Example: {@link org.geotoolkit.geometry.Envelope2D} and {@link org.geotoolkit.geometry.GeneralEnvelope}.

    Classes that are not GeoAPI implementation are usually not prefixed by any of the above.

  • Logging

    Geotoolkit uses the {@linkplain java.util.logging Java logging} by default, but can redirect its logging to Log4J or Apache's common-loggins. See {@link org.geotoolkit.util.logging} for details.
